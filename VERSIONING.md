# 版本、提交与 Release 规范

本规范是项目版本管理、Git 提交和公开发布的统一规则，适用于开发仓库、公开发布仓库
以及项目后续建立的其他仓库。构建、验证、生成产物、修改文档或更新发布仓库，都不会
自动授权 Git 提交、推送或创建 Release。

> 生效说明：本规范建立前已经正式发布的版本保持原样，不为了统一格式而回改历史；
> 本规范从下一个开发周期开始完整执行。

## 1. 版本开发周期

一个版本允许包含多个 commit。版本号表示这一组提交共同归属的开发周期，不要求每个
commit 都增加版本号。

例如，最新正式 Release 为 `v1.2` 时，后续开发进入 `v1.3`，在 `v1.3` 正式发布前
可以依次存在：

```text
v1.3：更新译文
v1.3：修正文档
v1.3：调整构建基线
v1.3：完成发布前验证
```

这些 commit 都属于同一个 `v1.3` 开发周期。只有 `v1.3` 正式发布以后，后续修改才
进入 `v1.4`。

版本递增遵循当前项目的一位小数版本规则：

```text
v1.2 -> v1.3
v1.3 -> v1.4
...
v1.9 -> v2.0
```

不得因为同一版本中增加了第二个、第三个 commit，就提前切换到下一版本。

## 2. commit 与 Release 的对应关系

每个正式 Release 必须对应该版本开发周期中最新且完整验证通过的 commit。

```text
v1.3 commit
    |
v1.3 commit
    |
v1.3 最终验证 commit  <-- v1.3 tag / v1.3 Release
```

发布时必须满足：

1. 该版本计划包含的修改已经全部完成。
2. 工作区没有遗漏的未提交修改。
3. 自动测试、完整构建、独立验证和发布产物检查全部通过。
4. 发布 ZIP 确实由准备打标签的最终 commit 构建。
5. Release 标签精确指向该版本最新的已验证 commit。
6. Release 附件的大小和 SHA-256 与本地最终产物一致。

不得把标签指向同一版本的较早 commit，也不得在正式发布后继续追加属于该旧版本的
commit。

## 3. 正式 Release 冻结

Release 一旦正式发布，其内容立即冻结，包括：

- Git tag 及其指向的 commit
- Release 标题
- Release 正文和更新摘要
- 已上传的 ZIP、安装包及其他附件
- 附件文件名、大小和内容

正式发布后不得编辑 Release 说明，不得替换、删除或重新上传附件，也不得移动或重建
同名标签。

如果发布后发现译文、文档、安装说明、Release 说明或附件存在问题，应当：

1. 保留旧 Release 原样。
2. 将修正纳入下一个版本。
3. 使用递增后的版本号创建新的 commit、tag 和 Release。

不得通过回改已经发布的 Release 来隐藏或覆盖历史。

Release 草稿在正式发布前可以检查和修正；草稿一旦发布，就适用上述冻结规则。

## 4. README 与 Release 的内容边界

README 只保存长期有效的项目介绍、功能、安装方法、更新方法、卸载方法、兼容性说明、
故障排查、许可和反馈渠道。

README 不应包含：

- “当前版本”一类会随每次发布失效的文字
- `Version Notes / 更新摘要`
- 某个具体版本的变更列表
- 某个 Release 专属的验证结果、附件校验值或升级摘要

每个版本的更新说明只写在对应 Release 中：

- v1.1 的更新摘要只属于 v1.1 Release。
- v1.2 的更新摘要只属于 v1.2 Release。
- 后续版本以此类推。

已经发布的旧 Release 说明保持不变，不为了统一格式而回头修改。

## 5. commit、push 与发布必须分别授权

以下三项是彼此独立的操作，必须分别获得作者明确授权：

1. 创建 commit
2. push 到远端仓库
3. 创建并正式发布 Release

“修改”“修复”“更新”“构建”“验证”“生成 ZIP”“更新发布项目”等指令，只授权完成
对应工作，不自动包含 commit、push 或发布授权。

执行规则：

- 没有明确要求“提交 commit”时，修改只能保留在本地工作区。
- 没有明确要求“push”时，即使已有 commit，也不得推送。
- 没有明确要求“发布 Release”时，即使已经推送并生成 ZIP，也不得创建 Release。
- 不得把一次授权扩大解释为后续版本或其他项目的长期授权。
- 这些规则适用于开发仓库、公开发布仓库以及其他所有项目。

完成工作后必须明确报告：

- 哪些文件已修改
- 是否已暂存
- 是否已 commit
- 是否已 push
- 是否已创建 Release

## 6. 提交信息

同一开发版本内的所有提交信息都使用该版本前缀：

```text
v<开发版本>：<本次提交摘要>
```

例如，最新正式 Release 为 `v1.2`、当前开发版本为 `v1.3` 时：

```text
v1.3：修正 README Markdown 格式
```

不得继续使用已经正式发布并冻结的 `v1.2：` 前缀。

## 7. 标准发布顺序

1. 确认最新正式 Release。
2. 将开发版本设为最新 Release 加 `0.1`。
3. 在同一开发版本下完成一个或多个 commit。
4. 完成最终测试、构建、验证和产物一致性检查。
5. 确认准备发布的 commit 是该版本最新 commit。
6. 在获得明确授权后 push。
7. 在获得明确授权后，为最终 commit 创建版本 tag 和 Release。
8. 一次性写好 Release 更新摘要并上传最终附件。
9. 从 GitHub 端反查 tag、正文、附件大小和 SHA-256。
10. Release 正式发布后冻结，不再修改。
